# AGENTS.md — Provenance Guard (AI201 Project 4)

Instructions for any AI coding agent (Cursor, Claude Code, GitHub Copilot, etc.) working in this repo.

## Source of truth

Read `planning.md` (repo root) in full before making design or scoring-logic changes. It is the spec for the detection signals, uncertainty/labeling rules, appeals workflow, and architecture — not this file. Don't duplicate design detail here; link to it instead.

## Python environment — always use `uv`

This project uses [`uv`](https://docs.astral.sh/uv/) with a committed `pyproject.toml` + `uv.lock`. **Never invoke a bare `python` / `python3` / `pip` / `pip3`** — that resolves to whatever interpreter happens to be on `PATH` (often the OS's system Python), not this project's `.venv`, and will silently install packages or run scripts against the wrong environment.

Always run:

```bash
uv sync                        # install/update deps into .venv
uv run app.py                  # run the app
uv run tools/calibrate.py      # run any script
uv add <package>               # add a new runtime dependency
uv add --dev <package>         # add a dependency only tools/ needs
```

If you add a new dependency, add it via `uv add`/`uv add --dev` (which updates `pyproject.toml` + `uv.lock`) rather than hand-editing `requirements.txt`. `requirements.txt` is kept only for non-`uv` consumers and should be treated as secondary — `pyproject.toml`/`uv.lock` are canonical.

## Project layout

Keep this separation when adding files — don't drop new scripts loose at the repo root:

| Path | What goes here |
|---|---|
| `app.py`, `signals.py`, `llm_signal.py`, `labels.py`, `submissions.py`, `audit_log.py` | Core runtime modules, imported by the running Flask app |
| `tools/` | Dev-only utilities never imported by `app.py` (e.g. `calibrate.py`, `planning_diagrams.py`, `try_signals.py`, `check_determinism.py`) |
| `tests/` | `pytest` suite — unit tests for `signals.py`/`audit_log.py`, integration tests for `app.py`'s routes |
| `planning.md` (repo root) | The design spec — source of truth, see above |
| `diagrams/` | `*.png` charts regenerated by `tools/planning_diagrams.py`, referenced from `planning.md` |
| `fixtures/` | Calibration/test text samples (`ai/`, `human/`) + the scripts that fetch/generate them |

## Running tests

```bash
uv run pytest             # whole suite
uv run pytest tests/test_signals.py -v
```

`tests/conftest.py` provides a `client` fixture (Flask test client with rate limiting disabled) so most tests aren't flaky against the real 10/minute `/submit` limit; `TestRateLimit` in `tests/test_app.py` re-enables it deliberately to test that behavior.

Scripts in `tools/` live one directory below the repo root, so they resolve paths via `Path(__file__).parent.parent` (see existing scripts for the pattern) and import root-level modules (e.g. `signals.py`) by inserting the repo root onto `sys.path` — follow that existing pattern rather than moving core modules into `tools/`.

## matplotlib / plotting scripts

Any script that generates figures (like `tools/planning_diagrams.py`) must force a headless backend before importing `pyplot`:

```python
import matplotlib
matplotlib.use("Agg")
import matplotlib.pyplot as plt
```

Without this, macOS's default `macosx` backend can abort the whole process when it tries to open a GUI window in a context with no WindowServer session (e.g. launched by an agent/editor subprocess) — this already happened once. These scripts only ever call `savefig`, never `show`, so there's no reason to risk a GUI backend.

## Secrets

Never commit `.env` or print/log its contents (it holds `GROQ_API_KEY`). Use `.env.example` as the template for required variables.
